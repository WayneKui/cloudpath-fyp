01
Overview

What is CloudPath?

CloudPath is a multi-cloud attack path engine. It scans your AWS and GCP accounts for security misconfigurations, builds a graph of how everything in your cloud connects, then correlates the misconfigurations into attack paths — chains of weaknesses an attacker could exploit to move from the internet to your sensitive data.

Most security tools give you a flat list of hundreds of issues. CloudPath answers the more useful question: "out of all these issues, which combinations actually let an attacker reach our data?"

What you get

🎯

MITRE ATT&CK Paths

Attack paths mapped to the standard adversary framework.

📊

CVSS Risk Scores

Transparent 0–10 scores so you know what to fix first.

🔗

Interactive Graph

Each path drawn as a live visual you can explore.

⚙

Open YAML Rules

Every rule is readable, editable, and extensible.

Scope

CloudPath analyzes cloud configuration (posture) — what's set up, what permissions exist, what's exposed. It does not monitor live activity or runtime behavior. Runtime monitoring is identified as future work.

02
Getting started

Quick Start

Get from zero to a working attack path scan in four steps. The whole journey takes about ten minutes.

STEP 01

Connect

Deploy our role template in your cloud account, paste back the Role ARN or key.

STEP 02

Open Scanner

Click Run Scan. Prowler and Cartography do their work in sequence.

STEP 03

Read Paths

The graph and panel show each attack path with steps, techniques, and scores.

STEP 04

(Optional) Rules

Add or edit detection rules. The engine picks them up automatically.

Connecting Clouds

CloudPath uses read-only access. For AWS, you deploy a CloudFormation template that creates an IAM role trusting CloudPath — no access keys are ever shared. For GCP, you provide a read-only service account key with the Viewer and Security Reviewer roles.

Head to the Connect page for the click-by-click flow. It takes about 5 minutes per cloud and is a one-time setup.

Running a Scan

From the Scanner page, clicking Run Scan does three things in sequence:

🔎

Prowler

Checks hundreds of misconfigurations and outputs JSON findings.

💾

Cartography

Ingests your cloud resources and relationships into Neo4j as a graph.

🧠

Engine

Merges findings with the graph and applies detection rules.

🔗

Chain

Links matched techniques into MITRE-mapped attack paths with CVSS scores.

Performance

A full scan against a real cloud account typically takes 2–5 minutes. The button shows a live scan-sweep so you know it's working.

Reading the Results

The Scanner page shows your paths in two views: an interactive graph on the left where nodes are colored by MITRE tactic and connected by red arrows showing attacker movement, and a detail panel on the right with each path's CVSS score, severity label, exploitability, impact, and step-by-step kill chain.

03
Under the hood

How the Engine Works

Three layers work together, each doing one job well:

🔎
Layer 1

Prowler

Multi-cloud misconfiguration scanner. Identifies what's wrong with each resource individually. Hundreds of CIS-aligned checks shipped out of the box.

💾
Layer 2

Cartography

Open-source mapper that loads your cloud's resources and relationships into a Neo4j graph database. How everything connects.

🧠
Layer 3

Engine

Loads your detection rules, finds MITRE techniques in the graph, then chains them into attack paths with DFS + BFS reachability.

Detection Rules

A detection rule is a YAML file in the /rules folder. Each rule says "this MITRE technique exists when this Cypher pattern matches a node in the graph." Example:

T1190_public_ec2_entry.yamlYAML 
id: T1190
mitre_name: Exploit Public-Facing Application
tactic: Initial Access
severity: high
cloud: aws
cypher: |
  MATCH (i:EC2Instance)-[:MEMBER_OF_EC2_SECURITY_GROUP]->(sg:EC2SecurityGroup)
  MATCH (sg)<-[:MEMBER_OF_EC2_SECURITY_GROUP]-(rule:IpPermissionInbound)
  WHERE rule.fromport <= 22 AND rule.toport >= 22
  RETURN DISTINCT i.id AS node_id, 'EC2Instance' AS node_type

The engine loads every YAML file in the folder. To add a rule, drop a file in directly or use the Rules page builder.

Chaining and Scoring

Once individual techniques are detected, the chainer links them by:

  • Shared node — two techniques on the same resource chain together
  • Graph reachability — a privileged compute node whose role can reach a target storage bucket through the policy graph chains to that bucket

The chainer is a depth-first traversal, respecting MITRE tactic order so paths flow forward (Initial Access → Privilege Escalation → Impact).

Why CVSS?

CVSS is the industry-standard open framework maintained by FIRST and used by commercial platforms like Wiz and Orca. Using it makes CloudPath's scores transparent, comparable, and defensible.

04
Make it yours

Writing Custom Rules

CloudPath ships with a starter set of rules, but its real value is that you can write your own. There are two ways — pick the one that matches you.

🔎
For everyone

Guided Mode

  • Identity: Technique ID, MITRE name, tactic, severity, cloud
  • Resource type: the kind of node to detect
  • Condition: public, has finding, attached policy, open port, principal
  • Value: e.g. 22 for port, * for any principal

The system translates those choices into valid Cypher behind the scenes.

⚙
For power users

Advanced · Cypher

  • Full access to all node labels, properties, and relationships
  • Must return node_id and node_type
  • Optional detail for human-readable context
  • Start from one of four built-in templates

Use when guided conditions aren't expressive enough for your detection.

Here's a real Cypher rule from the starter set:

T1530_public_s3_bucket.yamlCYPHER 
// Detect a public S3 bucket via its policy statement
MATCH (b:S3Bucket)-[:POLICY_STATEMENT]->(s:S3PolicyStatement)
WHERE s.principal CONTAINS '*' AND s.effect = 'Allow'
RETURN DISTINCT b.name AS node_id,
                'S3Bucket' AS node_type,
                s.action AS detail

The Rules page has template buttons for common patterns: public storage, admin role attached, open inbound port, cross-cloud credential. Use them as starting points.

Test before you trust

New rules should be tested against a known dataset before being trusted. Run a scan, confirm the rule fires when it should, and that it doesn't produce false positives on resources you know are safe.

Common questions

FAQ

+ Does CloudPath store my cloud credentials? ▾
For AWS, no — CloudPath uses cross-account role assumption, which only requires your Role ARN (a public identifier). Temporary credentials are generated per scan and expire within an hour. For GCP, the service account key file is stored encrypted and used only for read-only API calls.
+ Can CloudPath modify my cloud? ▾
No. The role and service account both use read-only permissions (SecurityAudit, ReadOnlyAccess, Viewer, Security Reviewer). CloudPath analyzes posture; it does not remediate or change anything.
+ How is this different from AWS Security Hub or GCP Security Command Center? ▾
Native tools cover a single cloud and present findings mostly as flat lists. CloudPath is multi-cloud, correlates findings into MITRE-mapped attack paths, and its rules are open and editable. Where native tools tell you what's wrong, CloudPath shows you how an attacker could chain those weaknesses into a real attack.
+ Why YAML rules and not a UI database? ▾
Open YAML files are readable, diffable in Git, easy to share, and impossible to lock you in. The Rule Manager page is just a convenience layer that writes the same YAML files you could write by hand.
+ How accurate is the risk score? ▾
The score follows CVSS principles (FIRST.org), the same foundation used by Wiz and Orca. It is heuristic — context and judgment matter. CloudPath shows the score's breakdown (exploitability, impact, length, internet entry) so you can see why a path scored as it did, and recalibrate if needed.
+ Can I run CloudPath without GCP? ▾
Yes — AWS-only scans work fine. GCP support is additive and only activates if you've connected a GCP project.
